# Codex 本地书库使用指南

这套书库适合把多本书交给 Codex 检索，再让 Codex 依据书中原文回答、解释或比较。首次设置完成后，导入、查询、解释和保存笔记都留在 Codex 对话窗口中；不需要自己运行检索命令，也不需要为了日常问答打开 Obsidian。

## 首次设置

按下面四步完成首次设置：

1. 在终端运行精确的克隆命令，然后进入项目目录：

   ```bash
   git clone https://github.com/zyf2238070278-glitch/codex-obsidian-book-library.git
   cd codex-obsidian-book-library
   ```

2. 在 Codex 中打开并信任整个项目，新建任务并说：“请安装并检查这个书库”。
3. 安装成功后，完整退出并重启 Codex，再打开同一项目。
4. 重启后新建任务并说：“检查书库状态”。

当前版本支持 macOS 16 或更高版本的 Apple Silicon（M 系列）Mac。首次安装需要联网，会下载约 500 MB 的语义模型及 Python 包。项目自带固定版本的 uv 和 Apple Vision 工具，在项目内创建项目本地 Python，按锁文件安装锁定版本的依赖，并准备语义模型和 OCR 模型；本机无需预装 Homebrew、Python、Xcode 或 uv。

安装器会在发布配置前检查依赖、384 维语义向量、Apple Vision、RapidOCR 和 MCP 的 `library_status`。用户不需要手工运行 Python、pip、uv 或模型下载命令。安装完成后，语义检索和 OCR 在本机运行；和 Codex 对话仍需正常联网。

书库的 SQLite 索引、语义向量和本地模型保存在项目的 `data/`，项目本地 Python 和依赖保存在 `.venv/`；这些运行数据不会写入 Obsidian。以后一般不必重复安装。移动项目目录后，在 Codex 中打开并信任新位置，重新运行“请安装并检查这个书库”，成功后再次完整退出并重启 Codex。修复或移动时重新运行不会删除已有书籍或笔记。

默认 Vault 位于项目的 `Obsidian书库/`。删除整个项目目录可能一并删除默认 Vault 中的原书、解析文本和读书笔记，删除、迁移或换电脑前请先备份。

当前支持的文件类型是：

- PDF（文件扩展名 `.pdf`）
- EPUB（文件扩展名 `.epub`）
- Markdown（文件扩展名 `.md`）
- TXT 纯文本（文件扩展名 `.txt`）

不支持的格式可以先转换为上述格式。扫描图片组成的 PDF 还需要 OCR，处理方式见“扫描版 PDF”。

## 在 Codex 中上传书籍

把书籍作为附件拖入 Codex 对话窗口，然后直接说：

> 把我刚上传的书导入书库，书名是《供应链入门》，作者是张三。

Codex 会把附件的本机绝对路径交给书库。导入完成后，可以继续在同一窗口说：

> 列出书库中的全部书籍和当前状态。

> 检查书库状态，告诉我哪些书需要处理，以及应该怎么恢复。

正常可检索的书通常是 `ready`。若有异常，Codex 会调用 `library_status` 查看具体错误和恢复建议：

- `keyword_only`：语义索引当前不可用，但关键词检索仍可用。先查看 `library_status` 返回的 `error`；旧记录若没有 `error`，按其中的保守建议检查模型状态和具体原因。
  - 若 `error` 含“语义模型未启用”或“缓存缺失”：重新运行“请安装并检查这个书库” → 完整退出并重启 Codex → 重新导入同一文件；完成后会补建语义向量。
  - 若 `error` 显示“语义索引失败”：先按 `error` 修复向量生成、模型运行或数据库问题，再重新导入同一文件；此时不要把问题直接归因于模型缓存。
- `needs_ocr`：PDF 几乎没有可提取文字。先做 OCR，生成可搜索的新 PDF，再在 Codex 中导入 OCR 后的文件。
- `failed`：解析或本地依赖出错。让 Codex 查看 `library_status` 返回的 `error`，修复对应问题后重新导入同一文件。
- `duplicate`：相同内容已经在书库中，不会再复制一份。这不是故障；让 Codex 列出书库或检查返回的书籍编号，继续使用已有记录即可。

若导入长时间停在 `processing`，也可以说“检查这本书为什么还在处理中，并按状态建议恢复”。不要反复盲目上传；先让 Codex 用 `library_status` 确认原因。

## 引用原文

需要逐字核对时，明确说“引用原文”，并尽量指出书名、主题和所需范围。例如：

> 在《供应链入门》中查找“牛鞭效应”的定义，引用两段最相关的原文，并标明出处。

> 找出作者谈需求预测失真的原话；只给必要的短引文，不要贴整章。

Codex 会先检索短预览，再展开少量最相关段落。回答中的逐字内容应标成“原文引用”，并附书名、位置和 `passage_id`，方便再次定位。

位置标记因格式而异：PDF 使用 **PDF 阅读器页码**（从阅读器看到的文件页序开始计数），它可能与纸书印刷页码不同；EPUB 没有稳定的跨设备页码，因此使用 **EPUB 章节**或小节名称。Markdown 和 TXT 通常使用标题、小节及 `passage_id`。引用前如果证据不足，Codex 应直接说明，而不是凭模型记忆补写书中原话。

为了减少 token 消耗，优先索取一至三段短引文。只有确实需要时再扩大范围，不要一次要求输出整本书或整章原文。

## 通俗解释

想理解含义而不是逐字抄录时，可以直接说：

> 先核对《供应链入门》里的相关段落，再用没有专业背景的人也能懂的话解释牛鞭效应。把书中观点和你的推断分开。

> 用一个生活例子解释作者这段话，同时保留出处。

Codex 应先取证，再把回答分成“书中原意的通俗转述”和必要的“Codex 推断”。转述不是原文，会明确标注；关键结论仍应带书名、PDF 页或 EPUB 章节及 `passage_id`。

跨书比较也可以留在同一个对话窗口：

> 比较《供应链入门》和《库存周期》如何解释需求波动。每本书分别给出依据，最后用通俗语言总结相同点和差异；只展开最相关的少量段落。

范围较大时，Codex 会分批进行有界检索，而不是把所有书一次塞进上下文。若某一本书没有足够证据，会单独指出。

## 保存到 Obsidian

只有你明确要求时，Codex 才会写入 AI 读书笔记。例如：

> 把刚才关于牛鞭效应的解释和引文保存到 Obsidian，标题叫《牛鞭效应：原文与通俗解释》；保留出处。

笔记会写入当前 Obsidian 仓库的 `书库/30-AI读书笔记/`，并标明它是 AI 生成内容。AI 笔记方便阅读，但不是原始证据；后续引用书籍时，Codex 仍应回到原书段落核对。

当前 Obsidian 仓库是安装时选择的 Vault；未指定时，就是项目内的 `Obsidian书库/`，不需要切换仓库。平时不需要操作 Obsidian；想浏览时，原书在 `书库/10-原始书籍/`，解析文本在 `书库/20-解析文本/`，AI 笔记在 `书库/30-AI读书笔记/`。

## 扫描版 PDF

扫描版 PDF 的每一页可能只是图片，没有可检索文字。书库会尽量识别这种情况并标记为 `needs_ocr`，但不会自动启动 OCR，也不会假装已经读到了图片中的文字。

只有你明确授权后才会处理。例如：

> 这本 PDF 显示 needs_ocr。开始 OCR 这本书，完成后检查状态。

Apple Vision 与 RapidOCR 都在本机运行，不上传原 PDF。查看、暂停、恢复和批量授权可分别说“查看 OCR 进度”“暂停 OCR”“继续 OCR”“处理所有待 OCR 书籍”。OCR 完成后让 Codex 调用 `library_status` 确认状态。OCR 可能识别错字，重要引文仍应回看对应 PDF 阅读器页码。

## 隐私边界

完整书籍和解析文本保留在本机的当前 Obsidian 仓库，SQLite 索引、语义向量和本地模型保留在项目的 `data/`；它们不会为了检索而整本发送给 Codex。普通流程只把检索预览和随后**选中少量段落**放入 **Codex 上下文**，供当前回答引用或转述。这种按需取证既让上下文更充裕，也避免为无关章节消耗大量 token。

但这不等于“书籍内容永远不会进入在线服务”：凡是进入 Codex 上下文的预览、段落，以及你主动粘贴或上传给 Codex 的内容，都应按你所使用的 Codex 服务及账户数据政策处理。不要导入你无权处理的书籍，也不要检索不应进入对话上下文的敏感内容。

希望进一步节省 token 时，可以这样说：

> 先只返回最相关的五条短预览；我选中后再展开最多两段原文。

> 优先通俗转述，只保留支持结论所需的短引文和出处。

这套书库的目标是“本地保存与检索、少量证据进入对话”，不是把十几本书永久放进模型上下文，也不是对模型本身进行微调。
