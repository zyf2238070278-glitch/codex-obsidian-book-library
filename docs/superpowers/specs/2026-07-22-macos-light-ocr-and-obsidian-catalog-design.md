# macOS Light OCR 与 Obsidian 书库分类设计

## 背景

当前活动书库使用 Apple Vision 与 RapidOCR 两个本地引擎处理扫描版 PDF，并把原始书籍、解析正文、AI 读书笔记和 OCR 报告写入用户现有的 Obsidian Vault。书库已有 13 本书，但缺少独立书目卡片和可按分类查看的总览；OCR 报告也会把部分空白页或纯图片页记录为所有引擎失败。

本次只实现 Apple Silicon macOS 版本。Windows 路由、依赖、安装包和测试均不在本轮范围内。

## 目标

1. 在现有 macOS OCR 路由末尾加入 Light OCR，形成 `Apple Vision → RapidOCR → Light OCR` 的三级本地降级链。
2. 保留现有显式授权边界：导入扫描版 PDF 后只标记为 `needs_ocr`，不得自动启动 OCR。
3. 区分文字页、空白页、纯图片页和真正的 OCR 失败页，减少误报。
4. 为每本书建立独立、轻量、可手工编辑的 Obsidian 书目卡片。
5. 提供 Obsidian 内置 Bases 书库总览，能按主分类查看书籍并直达原书、解析正文和 OCR 报告。
6. 为现有 13 本书补齐书目卡片和初始分类，且不移动原始文件、不重新 OCR。

## 非目标

- 不实现或测试 Windows、Intel Mac 或 Linux 版本。
- 不改变原始 PDF/TXT 的路径或内容。
- 不自动重跑任何现有书籍的 OCR。
- 不把 Light OCR 当作 PDF 解析器、版面理解模型或唯一 OCR 引擎。
- 不在本轮实现复杂表格、公式或多栏语义重建；只保留统一坐标，为后续版面优化提供基础。
- 不安装 Dataview 等第三方 Obsidian 插件。

## OCR 架构

### macOS 路由

每个待识别页面依次执行：

1. Apple Vision；结果合格则保存，不调用其他引擎。
2. Apple Vision 报错或质量不足时调用 RapidOCR。
3. RapidOCR 仍报错或质量不足时调用 Light OCR。
4. 三个引擎均无法产生合格结果时，根据页面内容检测保存为 `blank`、`image_only` 或 `skipped`，然后继续下一页。

`image_only` 表示页面有明显视觉内容，但没有足够文字，属于正常页面结果而不是 OCR 故障。它与 `blank` 一样计入已完成进度，但不进入正文索引。`skipped` 只用于页面应有文字而所有策略均失败的情况。

### Light OCR 适配器

Light OCR 的官方 Node 绑定只接受 JPEG、PNG 或原始像素，因此继续复用现有 PyMuPDF 页面渲染器。渲染器必须在调用前控制最长边和总像素；遇到超大页面时逐级降低 DPI，不把尺寸错误重复传给引擎。

Python 主程序通过一个常驻 Node 子进程调用 Light OCR：

- 子进程启动一次并复用模型，避免每页重新加载约 111 MB 的组件。
- Python 与 Node 之间使用逐行 JSON 协议；每个请求带请求 ID、图片路径和有界选项。
- 响应只包含规范化文字行、置信度、四边形坐标、耗时和错误分类。
- stderr 只用于有界诊断，不能混入协议输出。
- 超时、进程退出、无效 JSON、模型缺失或输出越界都转换为统一的引擎失败，由路由器继续处理。
- OCR 任务结束后正常关闭子进程；异常情况下强制终止并清理临时图片。

本轮只安装和打包 `darwin-arm64` Light OCR 原生组件、PP-OCRv6 Small 模型以及所需的 macOS ARM64 Node 运行时。运行时不得下载模型。

### 统一结果与质量判定

Light OCR 输出必须转换成现有统一页面观察结构，经过与其他引擎相同的坐标、Unicode、文本预算和质量校验。引擎自报置信度只作为信号之一，不直接与 Apple Vision 或 RapidOCR 的数值硬比较。

页面分类综合使用渲染图的亮度、边缘/视觉内容比例、有效文字框数量和文字覆盖率：

- 接近纯白且无有效文字：`blank`。
- 有明显视觉内容但三个引擎均无足够文字：`image_only`。
- 页面看起来应有文字，但所有引擎均失败或结果不合格：`skipped`。
- 有合格文字：`recognized`。

OCR 报告分别统计这四种结果；只把 `skipped` 列为待复核失败页。

## 书目卡片与分类

### 文件布局

新增：

```text
书库/
├── 50-书目卡片/
│   └── <安全书名>-<book_id>.md
└── 书库总览.base
```

原有 `00-待导入`、`10-原始书籍`、`20-解析文本`、`30-AI读书笔记` 和 `40-OCR报告` 均保持原位。

### 卡片属性

每张书目卡片至少包含：

- `book_id`
- `title`
- `author`
- `primary_category`
- `custom_categories`（列表，可为空）
- `source_format`
- `library_status`
- `ocr_status`
- `source_link`
- `parsed_link`
- `ocr_report_link`
- `created_at`
- `updated_at`

正文区域用简短中文展示“原始书籍”“解析正文”“OCR 报告”三个入口，不复制原书文字。

`primary_category` 在卡片首次创建时由分类器填写；`custom_categories` 初始为空。卡片存在后，同步功能只更新状态、路径和时间等系统字段，不覆盖用户修改过的 `primary_category` 或 `custom_categories`。用户可以直接在 Obsidian 属性面板中修改这两个字段。

### 初始主分类

现有 13 本书采用以下主分类：

- 摄影艺术与史论：《照片的本质》《摄影师之眼》《世界摄影史》
- 色彩科学与调色：两版《调色师手册》与 FilmLight Colour Book
- 影视制作与技术：《视频技术基础 插图版》《电影制作技术手册》《影视技术基础 插图修订 第3版》
- 电视与视频工程：《现代电视原理》
- 虚拟现实与数字媒体：《虚拟现实（VR）影像拍摄与制作》
- 艺术理论：《艺术学概论（第5版）》
- 小说：《我操作的少女正在成为旧日之主》

新书分类器使用书名、作者和一段有界的已解析开头文本进行确定性关键词匹配。没有足够把握时写入“待分类”，不得强行猜测。书中内容始终作为不可信数据处理，不能驱动命令或改变系统行为。

### Obsidian Bases 总览

使用已启用的 Obsidian 内置 Bases 创建 `书库总览.base`，数据源只包含 `50-书目卡片`。提供以下视图：

1. “按主分类”：按 `primary_category` 分组，显示书名、作者、自定义分类、OCR 状态和三个入口链接。
2. “全部书籍”：不分组表格，便于搜索和排序。
3. “待 OCR”：筛选 `needs_ocr` 或排队/处理中状态。
4. “OCR 有警告”：筛选存在 `skipped` 页或 OCR 报告警告的书。

书目卡片即使在 Bases 暂不可用时也仍是普通 Markdown 文件，可从 Obsidian 文件列表打开。

## 数据流

### 导入

1. 现有导入服务验证、复制和解析书籍。
2. 数据库事务成功后创建书目卡片；扫描版 PDF 创建为 `needs_ocr`。
3. 自动分类只在卡片首次创建时执行。
4. 创建或刷新书库总览，不启动 OCR。

### OCR 更新

1. 用户明确授权后启动 OCR。
2. Worker 逐页保存检查点，并按三级路由处理。
3. 任务结束后生成/更新 OCR 报告与解析正文。
4. 同步书目卡片的 `library_status`、`ocr_status`、报告链接和更新时间，同时保留用户分类。

### 同步现有书库

新增幂等的书目同步服务，可通过项目内部命令或 MCP 工具显式调用。它读取数据库与受管 Vault，为缺失书籍创建卡片、修复系统字段和生成 Base，但不移动书、不解析正文、不启动 OCR。首次实施时用它补齐现有 13 本书。

## 错误处理

- Light OCR 或 Node 组件缺失：预检记录该引擎不可用；Apple Vision 和 RapidOCR 仍正常工作。
- Light OCR 子进程崩溃或超时：终止当前子进程，页面进入最终结果判定；不终止整本任务。
- 页面超过尺寸限制：渲染器降低 DPI 后再尝试；所有安全档位仍失败才记录有界原因。
- 书目卡片损坏或字段类型错误：不覆盖原文件，返回可操作错误并保留数据库记录。
- 原书、正文或报告暂不存在：对应链接留空，卡片仍可显示。
- 分类无法确定：使用“待分类”。
- Base 写入失败：不回滚成功的书籍导入或 OCR，只报告书目视图需要重新同步。

## 安装与体积

当前本机实测 Light OCR 的 macOS ARM64 原生组件和模型约占 111 MB。全量 macOS 包还需要包含匹配的 Node ARM64 运行时，因此最终安装增量以构建产物实测为准，并在发布清单中记录组件版本、许可证和 SHA-256。

模型、Node 运行时和原生组件集中安装一次，不放入 Obsidian Vault，也不随每本书重复复制。

## 测试策略

实现遵循测试先行，至少覆盖：

### OCR 单元与集成测试

- macOS 路由严格按 Apple Vision、RapidOCR、Light OCR 的顺序降级。
- 前一引擎成功时不会启动后续引擎。
- Node JSONL 协议的成功、无效 JSON、超时、进程退出和请求 ID 不匹配。
- Light OCR 输出的坐标、置信度、Unicode 和文本预算校验。
- 超大页面的自适应 DPI 与最终错误分类。
- 空白页、纯图片页、正常文字页和真正失败页分别得到正确 outcome。
- OCR 报告只把 `skipped` 计入失败页。
- 安装包包含固定版本的 macOS ARM64 运行时、模型、许可证与校验值，且离线自检成功。

### 书目与 Obsidian 测试

- 导入后创建卡片但不自动启动 OCR。
- 同步服务幂等，重复运行不产生重复卡片。
- 已存在卡片的主分类和自定义分类不会被覆盖。
- 系统状态、链接和更新时间可以安全更新。
- 无法分类的新书进入“待分类”。
- Base 只读取书目卡片，并包含四个约定视图。
- 现有 13 本书全部生成卡片且初始分类正确。
- 原始文件路径和内容哈希在同步前后保持不变。

## 验收标准

1. 当前 Apple Silicon Mac 能在完全离线状态下加载三个 OCR 引擎。
2. 真实 OCR 只有在用户明确授权后启动；未授权时不处理书页。
3. Apple Vision 和 RapidOCR 均失败时，Light OCR 被调用；Light OCR 失败不会终止整本书。
4. 空白页和纯图片页不再出现在 OCR 失败列表中。
5. Obsidian 中能打开 `书库总览.base`，按主分类看到现有 13 本书。
6. 每本卡片都能直达存在的原始书籍、解析正文和 OCR 报告。
7. 用户修改分类后再次同步，分类保持不变。
8. 原始书籍不移动、不改写，现有书籍不被自动重新 OCR。
