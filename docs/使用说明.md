# 使用说明

日常操作都在 Codex 中用自然语言完成。Obsidian 用来浏览原书、解析文本和已经保存的笔记，不用在 Obsidian 中输入命令。

## 1. 先检查状态

每次刚安装、移动目录或更换 Vault 后，新建 Codex 任务并说：

> 检查书库状态。

如果显示 0 本但状态正常，只表示还没有导入书。后续新任务不必重复安装，也不需要每次输入 MCP 命令；只要继续打开并信任整个项目即可。

## 2. 导入一本书

在 Codex 对话框附加 PDF、EPUB、Markdown 或 TXT，然后说：

> 导入这本书。

等待 Codex 报告导入成功后再提问。不要仅把文件复制到 Obsidian 目录后就假定已经建好索引。如果导入失败，请按错误提示处理，不要把失败结果当成可检索书籍。

## 3. 列出书库

> 列出书库里的书，并告诉我每本书当前是否可检索。

这适合确认书名、重复导入和处理状态。

## 4. 扫描版 PDF：明确启动 OCR

导入扫描版 PDF 后，系统只保存原书并标记待 OCR，不会自动识别。确认要在本机处理后说：

> 开始 OCR 这本书。

需要查看进度时说：

> 查看 OCR 进度。

需要暂停或继续时说：

> 暂停 OCR

> 继续 OCR

要批量处理时说：

> 处理所有待 OCR 书籍。

OCR 在 Apple Silicon Mac 的本机 Apple Vision 上运行，不上传原 PDF，也不需要 OCR token。任务按页保存 checkpoint，可暂停、恢复；扫描质量、字体和版式可能造成识别错误。引用 OCR 内容前，请按原 PDF 的物理页核验，并把它当作待核对证据。

## 5. 请求原文短引用

> 根据《书名》回答这个问题：……。请引用两段短原文，并标出《书名》和 PDF 页；如果是 EPUB，标出 EPUB 章节。

原文引用应短到只够支撑结论。Codex 会先检索，再展开选中的段落；不会把搜索预览冒充完整原文。如果书库没有足够证据，它应明确说明，而不是凭模型记忆补成书中观点。

## 5. 请求通俗解释

> 根据《书名》，用通俗的话解释“……”，并把原文依据、通俗转述和 Codex 推断分开写。

“原文引用”来自核验过的书中段落；“通俗转述”是对这些段落的改写；“Codex 推断”是基于证据的进一步理解，三者不应混在一起。

## 6. 跨书比较

> 比较《书 A》和《书 B》对“……”的看法。分别给出证据位置，再总结相同点、不同点和 Codex 推断。

宽泛问题会拆成多次有界检索，再按证据汇总。需要更多细节时，最好限定主题或章节，避免一次要求覆盖整个书库。

## 7. 保存读书笔记

先阅读和修改当前回答；确认内容后再明确说：

> 把上面的结论保存为读书笔记，标题是“……”。

只有明确说“保存为读书笔记”或同义要求时，Codex 才会写入 `书库/30-AI读书笔记`。普通提问、解释或总结不会自动保存。保存的引用只应使用本次已经核验的段落。

## 为什么这样更省上下文

书籍导入后，系统按需检索与问题最相关的预览和短段落，不会把整本书塞进上下文。这既便于核对来源，也能节省 token。被选中的少量证据仍会进入 Codex 对话并参与回答，所以并不是零 token 或零内容传输。
