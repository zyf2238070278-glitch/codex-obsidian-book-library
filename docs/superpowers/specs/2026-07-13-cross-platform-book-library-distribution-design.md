# Codex Obsidian 书库助手跨平台分发设计

日期：2026-07-13  
状态：已完成方案讨论，待用户复核书面规格

## 1. 目标

把现有本地书库 RAG 项目制作成可公开分享的 GitHub 项目，让普通 Codex 用户在 macOS 或 Windows 上完成一次安装后，继续只在 Codex 对话窗口中导入书籍、检索原文、获得通俗解释并把读书笔记保存到 Obsidian。

本次交付同时提供：

1. 一个公开 GitHub 源码仓库，暂定仓库名 `codex-obsidian-book-library`。
2. 一个包含语义模型的全量 ZIP 安装包，供非技术用户直接下载。
3. macOS 和 Windows 的双击安装入口。
4. 中文《安装说明》《使用说明》《常见问题》《隐私与数据存放说明》以及第三方许可说明。

“下载就能使用”在本项目中的定义是：用户无需自己编写配置或输入 Python、uv、MCP 命令；解压并双击对应安装入口后，安装器自动准备运行环境。Codex 本身以及首次安装 Python 依赖仍需要互联网连接。全量 ZIP 已携带语义模型，不再单独从 Hugging Face 下载模型。

## 2. 支持范围

首个公开测试版支持：

- macOS 14 或更高版本，Apple Silicon。
- Windows 10/11，x64。
- Codex Desktop 本地项目。
- 可选连接一个已经存在的 Obsidian Vault；也允许使用项目内置 Vault。
- PDF、EPUB、Markdown 和 TXT 导入。

首版不承诺：

- Intel Mac、Linux、Windows ARM。
- 扫描 PDF 的自动 OCR。
- 无网络安装全部 Python 依赖。
- 自动迁移现有用户的 SQLite 数据库、绝对路径索引或旧书库。
- 把完整书籍、用户笔记或数据库发布到 GitHub。

## 3. 分发形式

### 3.1 GitHub 源码版

Git 仓库只包含源码、测试、安装程序、配置模板、空 Vault 目录和文档。它不得包含本机虚拟环境、语义模型、SQLite 数据库、书籍、解析文本、AI 笔记、个人用户名或绝对路径。

从 Git 克隆的用户运行安装入口时，安装器从本项目的 GitHub Release 下载固定版本的语义模型包，然后执行与全量 ZIP 相同的安装和自检流程。

### 3.2 全量 ZIP 版

GitHub Release 提供一个全量 ZIP，内含：

- 与对应 Git 标签一致的源码。
- macOS 与 Windows 安装入口。
- 固定 revision 的 `intfloat/multilingual-e5-small` 模型快照。
- 模型许可证、模型卡、文件清单和 SHA-256 校验值。

ZIP 不包含预构建 `.venv`。虚拟环境和二进制依赖与操作系统、CPU 和安装路径有关，必须由安装器在目标电脑上创建。

### 3.3 Release 与版本

- 使用语义化版本，例如 `v0.1.0`。
- Release 标记为测试版，直到至少一位外部 macOS 用户和一位外部 Windows 用户完成安装验证。
- 每个发布资产提供 SHA-256。
- 大型模型只进入 Release 资产，不进入 Git 历史或 Git LFS。

## 4. 安装体验

### 4.1 用户流程

1. 从 GitHub Release 下载全量 ZIP 并解压。
2. macOS 双击 `安装书库.command`；Windows 双击 `安装书库.bat`。
3. 安装器检测操作系统、架构、网络、磁盘空间和目录权限。
4. 安装器自动准备 `uv` 和受控的 Python 3.12 环境，并按锁文件安装依赖。
5. 安装器检测 Obsidian Vault。检测到一个明确 Vault 时请求确认；无法唯一确定时让用户粘贴或选择 Vault 路径；用户也可选择项目内置 Vault。
6. 安装器创建 `书库/00-待导入`、`10-原始书籍`、`20-解析文本` 和 `30-AI读书笔记`。
7. 安装器验证并复制随包提供的语义模型，生成本机专用 `.codex/config.toml`。
8. 安装器执行 SQLite FTS、384 维语义向量、六个 MCP 工具和 `library_status` 烟测。
9. 安装器给出单一成功或失败结论；成功时提示用户用 Codex 打开并信任当前文件夹，然后重新加载一次 Codex。

安装完成后，日常操作不再要求终端命令。用户只需上传书籍并说“导入”，或直接自然语言提问。

### 4.2 安装入口

- `安装书库.command`：macOS 可双击的薄包装，调用共享安装核心。
- `安装书库.bat`：Windows 可双击的薄包装，以适当的 PowerShell 执行策略调用共享安装核心。
- `installer/setup.py`：跨平台安装、配置生成和自检核心，所有可测试逻辑集中在这里。

平台包装只负责启动；路径处理、错误消息和配置生成不在 shell/批处理文件中重复实现。

### 4.3 错误处理

安装失败时保留完整日志，但界面只给出清晰的中文原因和下一步。必须覆盖：

- 系统或架构不支持。
- 磁盘空间不足。
- 无法准备 Python/uv 或安装依赖。
- Obsidian 路径不存在、不可写或不安全。
- 模型缺失、损坏或校验不符。
- SQLite 缺少所需 FTS 能力。
- MCP 无法启动或工具数量不符。

失败不得留下声称“已安装”的配置；重复运行安装器必须安全且可恢复。

## 5. 跨平台改造

现有实现依赖 `fcntl`、`flock`、`O_DIRECTORY`、`O_NOFOLLOW`、`dir_fd`、inode 身份和 POSIX 权限行为，Windows 无法直接启动。改造采用平台适配层，而不是在业务代码中散布系统判断。

新增或重构一个文件系统安全层，负责：

- 进程间锁：macOS 使用 `fcntl`，Windows 使用可用的 Windows 文件锁机制。
- 原子发布和替换：优先使用同文件系统原子重命名，并为 Windows 的占用和替换语义提供有界重试。
- 路径与重解析点校验：在各平台阻止受管目录逃逸、意外符号链接或 Windows reparse point。
- 文件身份和重复导入保护：保留现有安全目标，但使用各平台可验证的能力实现。
- SQLite 与模型目录的安全创建和权限检查。

业务层继续通过稳定接口处理导入、解析、索引和保存笔记。平台差异必须有独立单元测试；macOS 与 Windows 都要运行端到端测试。

## 6. 配置与数据位置

仓库中的 `.codex/config.toml` 改为模板或由安装器生成，不再提交任何人的绝对路径。生成的配置必须：

- 使用目标电脑虚拟环境中 Python 可执行文件的绝对路径，不依赖 Codex GUI 的 shell `PATH`。
- 设置本机项目根路径和用户选择的 Obsidian Vault。
- 强制语义模型运行时离线加载。
- 只启用六个书库 MCP 工具。

数据边界：

- Obsidian：原书、解析文本、AI 读书笔记。
- 项目本地 `data/`：SQLite 索引、向量、模型运行副本和锁文件。
- GitHub：源码、空目录模板、安装器、测试与文档。

`data/`、`.venv/` 和受管书籍目录始终被 Git 忽略。发布前执行自动隐私扫描，拒绝包含 `/Users/`、Windows 用户目录、当前用户名、`.sqlite3`、书籍文件或 AI 笔记的发布包。

## 7. 模型与许可证

- 使用 `intfloat/multilingual-e5-small`，固定具体 revision，不跟随浮动的 `main`。
- 分发展开后的、可独立读取的模型目录，不依赖容易在 ZIP 中损坏的 Hugging Face 缓存符号链接。
- 随包附带模型卡、MIT 许可证、来源链接、固定 revision 和 SHA-256 清单。
- 项目源码采用 MIT 许可证，便于公开分享和二次测试。
- `THIRD_PARTY_NOTICES.md` 列出模型和随项目直接分发的第三方内容。

## 8. Codex 与 Obsidian 行为

项目保留并泛化 `AGENTS.md` 的证据规则：

- 书库相关声称先检索。
- 引用或转述前展开已选择段落。
- 明确区分原文、通俗转述和 Codex 推断。
- AI 笔记不得作为原书证据。
- 只有用户明确要求时才保存笔记。
- 保持预览、段落和来源 token 上限。

安装完成后的最短使用路径：

1. 把 PDF/EPUB/MD/TXT 拖入 Codex。
2. 说“导入这本书”。
3. 直接问“根据《书名》解释某主题并标出处”。
4. 需要时说“引用两段短原文”或“保存到 Obsidian”。

## 9. 文档

仓库至少包含：

- `README.md`：项目简介、下载按钮、三分钟开始、支持平台和测试版提示。
- `docs/安装说明.md`：macOS 与 Windows 分栏说明、Obsidian 选择、Codex 信任/重新加载、卸载和升级。
- `docs/使用说明.md`：导入、查询、引用、通俗解释、跨书比较和保存笔记的示例。
- `docs/常见问题.md`：工具不显示、路径问题、`keyword_only`、`needs_ocr`、超时和重复导入。
- `docs/隐私与数据存放.md`：本地保存范围、会进入 Codex 上下文的少量证据以及公开仓库不包含的内容。

文档面向没有 Python、Git 或 MCP 背景的用户，不把“自己改配置”作为正常安装步骤。

## 10. 测试与发布门槛

本地和 CI 必须覆盖：

- 现有 Python 测试套件。
- 安装器配置生成、幂等重跑、失败回滚和路径转义。
- macOS 与 Windows 文件锁、原子发布、重复导入和受管目录安全性。
- PDF、EPUB、Markdown、TXT 的最小端到端导入与搜索。
- 模型文件清单、SHA-256、离线加载和 384 维向量烟测。
- MCP 六个工具注册和 `library_status`。
- GitHub Actions 的 macOS 与 Windows 测试矩阵。
- 发布包解压后的清洁安装烟测。
- 发布包个人路径、数据库、书籍和笔记泄漏扫描。

只有 macOS 本地测试通过、Windows GitHub Actions 通过、全量 ZIP 校验通过且隐私扫描为零命中时，才创建公开测试版 Release。

完成后用户获得：

1. 公开 GitHub 仓库链接。

4. 安装说明和使用说明的直接链接。
5. 发布版本号、SHA-256 和已验证平台说明。
