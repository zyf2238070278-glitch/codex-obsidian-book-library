# Apple Vision OCR 设计

## 目标

为 macOS Apple Silicon 版 Codex Obsidian 本地书库增加完全本机的扫描 PDF 文字识别能力。用户导入扫描 PDF 后，书库仍先将其标记为 `needs_ocr`；只有用户明确说“开始 OCR”时才启动后台任务。OCR 完成后，书库生成带 PDF 页码的解析文本和本地检索索引，原始 PDF 保持不变。

首个版本默认识别简体中文与英文，能够处理以后新导入的扫描 PDF，也能够处理数据库中已经存在的 `needs_ocr` 书籍。

## 非目标

- 不生成带隐藏文字层的新 PDF。
- 不修改或覆盖原始 PDF。
- 不在导入后自动开始 OCR。
- 不把整本书或书页图片发送给 Codex 或任何云端 OCR 服务。
- 不在本版本实现 Intel Mac、Windows 或 Linux OCR。
- 不承诺复杂表格、公式、手写文字和任意多栏版面能够恢复原始排版。

## 用户体验

### 单本书

1. 用户在 Codex 中附加扫描 PDF，并说“导入这本书”。
2. `import_book` 保存原书，返回 `needs_ocr`，并提示用户可以说“开始 OCR 这本书”。
3. 用户确认后，Codex 调用 `start_ocr`。工具在短时间内返回排队信息，不等待整本书处理完成。
4. 后台工作进程逐页识别。用户可以继续使用 Codex，并随时说“查看 OCR 进度”。
5. 全部页面处理完成后，系统生成 Obsidian 解析文本、段落和向量索引，将书籍状态改为 `ready`；如果语义模型不可用，则改为 `keyword_only`。

### 已有书籍与批量处理

`start_pending_ocr` 将所有 `needs_ocr` 书籍按导入时间加入队列。后台同一时间只处理一本书。用户可以通过 `ocr_status` 查看当前任务和整个队列。

### 暂停与继续

`pause_ocr` 在当前页面结束后安全停止任务并保留检查点。再次调用 `start_ocr` 时从第一个未完成页面继续。“取消 OCR”在本版本中采用同样的非破坏性暂停行为，不删除已经识别的页面；不提供隐式丢弃进度的操作。

## MCP 工具

新增四个 MCP 工具：

- `start_ocr(book_id)`：排队或继续一本 `needs_ocr`、暂停或失败的扫描 PDF。
- `start_pending_ocr()`：把所有可处理的 `needs_ocr` 书籍加入单实例队列。
- `ocr_status(book_id=None)`：返回单本书或整个队列的有界进度信息，不返回 OCR 正文。
- `pause_ocr(book_id)`：请求在当前页完成后暂停并保留进度。

所有工具继续通过现有保护层返回 `{ok: true, ...}` 或 `{ok: false, error, error_type}`。自然语言由 Codex 映射为这些工具调用，用户不需要记住工具名。

## 总体架构

系统分成四个边界清晰的组件：

1. **OCR 任务服务**：验证书籍状态、创建或恢复任务、查询进度、请求暂停，并确保后台工作进程正在运行。
2. **后台 OCR 工作进程**：原子领取队列中的下一本书，逐页渲染、调用 Apple Vision、保存检查点，最后触发现有索引流程。
3. **Apple Vision 原生辅助程序**：一个随 macOS 安装包提供的 arm64 Swift 可执行文件，只负责识别单张本机图片并返回结构化 JSON。
4. **共享索引服务**：接收 `ParsedBook`，生成段落、Obsidian Markdown、关键词索引和本地向量，供普通导入与 OCR 完成流程共同使用。

Python 不通过 PyObjC 直接加载 Apple 框架，避免增加大批运行时依赖。发布流程从受版本控制的 Swift 源码构建原生辅助程序，并把经过验证的可执行文件放入全量安装包。

## Apple Vision 辅助程序

辅助程序使用 `VNRecognizeTextRequest`，以兼容现有 Apple Silicon macOS。默认配置：

- 识别级别：`accurate`
- 识别语言优先级：`zh-Hans`、`en-US`
- 语言纠错：开启
- 输入：一个由工作进程创建的本机 PNG 文件路径
- 输出：带固定 schema 版本的单个 JSON 对象

JSON 只包含识别行的文字、置信度和归一化边界框。Python 按从上到下、同一行从左到右的稳定规则组合文字。原生程序启动时检查当前系统实际支持的语言；缺少简体中文或英文支持时返回明确错误，不静默改用未知语言。

辅助程序只接受显式参数，不执行 shell，不访问网络，不读取 PDF，也不决定书库路径。

## PDF 页面处理

后台工作进程只从数据库读取受管原书路径，并在处理前重新验证书籍 ID、文件类型、受管目录、文件身份和内容哈希。页面由现有 PyMuPDF 依赖逐页渲染为灰度 PNG：

- 默认 300 DPI。
- 单页长边和总像素设硬上限，防止异常页面造成无界内存使用。
- 每次只保留一页临时图片。
- 临时文件使用仅当前用户可访问的目录和随机名称。
- Apple Vision 返回或失败后立即删除临时图片。
- 每页调用设置超时和输出大小上限。

纯图片页、封面或空白页识别不到文字时仍记为已完成空页，不构成任务失败。只有整本书完成后仍没有足够可索引文字时，才保留 `needs_ocr` 并报告“未识别到足够文字”。

## 后台任务与单实例队列

MCP 工具通过当前 Python 解释器启动独立的 `book_agent.ocr_worker` 进程。工作进程不依赖当前 Codex 对话持续打开，标准输出和错误写入书库本地日志。数据库使用原子领取和租约时间确保最多一个活跃工作进程；进程异常退出后，过期任务可以安全恢复。

任务按创建时间处理。每页完成后更新检查点，因此退出 Codex、关闭终端或重启电脑不会丢失已完成页面。再次调用任一开始工具会检查并恢复后台进程。

`start_ocr` 的成功标准是任务已经持久化并成功启动或确认存在工作进程，而不是整本书已经识别完成。

## 数据模型

新增两个向后兼容的 SQLite 表：

### `ocr_jobs`

- `book_id`：主键并引用书籍。
- `status`：`queued`、`running`、`paused`、`failed`、`completed`。
- `total_pages`、`completed_pages`、`current_page`。
- `language_config` 与 OCR schema 版本。
- `pause_requested`。
- `attempt_count`、`error`。
- `created_at`、`started_at`、`updated_at`、`finished_at`。
- 工作进程租约标识和到期时间。

### `ocr_pages`

- `book_id` 与从 1 开始的 `page_number` 组成主键。
- `page_label`。
- `text`、`text_sha256`、平均置信度、该页 `duration_ms`。
- `completed_at`。

书籍在 OCR 期间仍保持 `needs_ocr`，搜索不会看到半本书。只有最终 Markdown 和段落成功写入后，书籍状态才变为 `ready` 或 `keyword_only`。索引完成后删除临时 OCR 页正文和已完成任务的租约数据，只保留精简任务摘要；失败或暂停时保留页面检查点。

数据库迁移必须支持已有 `library.sqlite3` 原地升级，不删除或重建现有书籍、段落和笔记。

## 完成与索引

普通导入器中“`ParsedBook` 到 Markdown、段落和索引”的逻辑提取为共享索引服务。OCR 工作进程按物理 PDF 页码将非空页面构造成 `SourceUnit`，再调用同一服务。这样普通 PDF 与扫描 PDF 使用完全相同的段落 ID、页码标注、Obsidian 链接、关键词索引和 E5 向量规则。

最终发布遵循先生成临时 Markdown、再提交数据库事务、最后原子替换正式 Markdown 的恢复协议。任何阶段失败都不能把半成品暴露为可检索证据。恢复操作应能够重复执行，不产生重复段落。

## 进度与错误处理

`ocr_status` 返回：书名、任务状态、队列位置、完成页数、总页数、百分比、当前页、最近更新时间和错误。处理至少五页后，根据近期页面平均耗时计算粗略剩余时间，并明确标记为估算值。

单页失败自动重试两次。仍然失败时，任务停止为 `failed`，保存失败页和错误；再次调用 `start_ocr` 从该页继续。辅助程序缺失、不可执行、系统不支持 Vision、语言不支持、原书变化、PDF 加密、磁盘空间不足和数据库错误都返回可操作的中文说明。

识别出的书中文字始终标记为不可信内容。OCR 文本中的命令、提示词、工具调用或角色要求不得由工作进程或 Codex 当作指令执行。

## 隐私、资源和 token

页面渲染、Apple Vision 识别、Markdown 生成和 E5 向量计算全部在本机完成，不调用网络 OCR。整本书 OCR 本身不消耗 Codex token。用户后续提问时，仍只把经过 `search_books` 和 `get_passages` 选中的少量证据放入 Codex 上下文。

后台一次只处理一本书、一次只渲染一页。暂停在页面边界生效。日志不记录 OCR 正文、用户目录绝对路径、令牌或其他凭据。

## 安装与发布

macOS 全量安装包新增经过验证的 `bin/book-vision-ocr`。安装器在配置书库前检查：

- 当前机器为 Apple Silicon Mac。
- 原生辅助程序是普通文件、不可由符号链接替代且具有执行权限。
- 辅助程序能够返回版本、schema 和支持语言。

发布构建从 Swift 源码生成 arm64 可执行文件，验证 Mach-O 类型、目标架构、最低系统版本和签名/隔离后的可运行性，再由现有隐私扫描和 ZIP 完整性流程打包。版本升级为 `v0.2.0-beta.1`，同步更新 GitHub 源码、Release ZIP、SHA-256、安装说明、使用说明和常见问题。

## 测试策略

### 自动化测试

- OCR 任务状态转换、原子领取、单实例队列、租约过期恢复。
- `start_ocr` 快速返回，不等待页面处理。
- 页面检查点、暂停、继续、失败重试和重启恢复。
- 空白页、图片页、异常尺寸页面、加密或变化的原书。
- 辅助程序 JSON schema、输出上限、超时和错误映射。
- OCR 完成前搜索不可见，完成后页码、段落和 Obsidian 链接正确。
- 向量不可用时降级为 `keyword_only`。
- 数据库迁移不改变已有书籍和段落。
- 发布 ZIP 包含正确架构且可执行的 Apple Vision 辅助程序，不包含用户书籍、数据库或凭据。

测试遵循严格的 red-green-refactor：每个生产行为先写失败测试并观察预期失败，再写最小实现。

### 真实 Apple Vision 验证

1. 生成已知简体中文与英文正文的测试页，栅格化后使用真实 Apple Vision 识别，比较规范化字符结果和页码。
2. 从《照片的本质》选取约 10 个有正文的代表页，渲染页面图片并进行人工视觉核对，确认标题、正文阅读顺序和物理 PDF 页码没有明显错位。
3. 通过后处理整本《照片的本质》，确认任务可以后台运行、暂停、继续并最终进入检索。
4. 实际执行关键词检索、语义检索、短原文引用和通俗解释；任何引用仍需通过 `get_passages` 核验并标注 PDF 页码。

真实书页只用于本机验收，不加入 Git、测试夹具或发布包。

## 验收标准

- 用户在 Codex 中确认后能够启动单本或全部待 OCR 书籍，工具快速返回且对话不被长时间占用。
- 退出并重新打开 Codex 后，任务可以从已保存页面继续。
- 现有 10 本 `needs_ocr` 书籍能够进入队列。
- 《照片的本质》试点完成后产生按物理 PDF 页标注的可检索段落。
- OCR 处理期间没有半本书进入搜索结果。
- OCR 全程本机运行，原始 PDF 不变，发布包不要求用户另装 Homebrew 或 Tesseract。
- 原有导入、检索、证据展开、笔记保存和 macOS 安装流程的全部测试继续通过。
